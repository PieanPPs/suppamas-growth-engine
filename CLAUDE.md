# Suppamas Growth Engine

EdTech platform for Anusorn Suppamas School. Next.js App Router + TypeScript + Supabase (PostgREST). User communicates in Thai; admin/teacher-facing app.

## Workflow (always follow)

1. Before pushing any change: `npx tsc --noEmit` then `npm run build` — both must be clean.
2. Commit + push after each completed fix/feature (established pattern, don't batch unrelated fixes into one commit).
3. Any schema-affecting change requires giving the exact SQL migration to the user in the same reply — they run it manually in Supabase SQL Editor (RLS blocks anon-key DELETEs; writes/updates work fine via the app, but destructive/DDL changes go through SQL Editor).
4. When context is getting long mid-session: proactively suggest `/compact`, or summarize + keep going, rather than waiting for it to auto-trigger. Keep this file's "Pending / open items" section current so state survives a compact or a new session.

## Critical gotchas

- **PostgREST hard-caps every request at 1000 rows server-side** — no client `.limit()` override works. Any table that can exceed 1000 rows must be read via `fetchAllPaged<T>()` in `src/lib/db.ts` (pages with `.range()` until a short page returns), not `.limit()`/`.select()` alone. Applies to `student_assessments`, `homework_submissions`, `trait_ratings`, `attendance`, `test_scores`, `component_scores`, `test_item_responses`, `pacing_logs`, `plan_submissions`.
- **Upserts, not check-then-insert-or-update.** Use `.upsert(payload, { onConflict: '...' })` for write paths where a row may or may not already exist (e.g. homework submissions keyed on `student_id,module_id,lesson_plan_id`). Manual "check local state then insert-or-update" logic is fragile against stale/incomplete local state.
- **Term-scoping analytics**: only apply `getTermStartISO()` filtering to tables that represent *per-term growing history* (`student_assessments`, `test_scores`). Do NOT term-scope `pacing_logs` / `plan_submissions` (cumulative/latest-wins state) or `lib/impact-data.ts` (all-time stakeholder metrics) — doing so makes prior-term completed work look incomplete.
- **Indicator grouping uses `standard`** (มาตรฐาน, e.g. "ค 1.1") not `strand` (สาระ, often unset) — matches the convention on `src/app/teacher/curriculum/page.tsx`.
- **Pacing status is per-lesson-plan**, not per-module: use `latestPacingByLessonPlan()` (keyed on `lesson_plan_id`) for lesson-plan cards, `latestPacingByModule()` (module-only rows, `lesson_plan_id IS NULL`) for module-level rollups. Don't conflate the two.
- **`student_assessments` is one record per (student, module, lesson_plan_id), not one per day.** Confirmed with user 2026-07-03. Read/write paths must key off `student_id,module_id,lesson_plan_id` (upsert on that triple, look up existing regardless of date) — never filter by `created_at`/today when checking "already assessed," that was the bug in the original `/teacher/assessment` page. Every page aggregating this table must pass rows through `latestAssessmentPerPlan()` from `src/lib/db.ts` before averaging/counting — audited + applied everywhere 2026-07-06.
- **`lesson_plans.planned_week`**: a module spans multiple weeks (`planned_week`..`planned_week + expected_duration_weeks - 1`); an individual lesson plan/topic can be taught in any week within that span. Legacy plans with `planned_week = null` fall back to displaying on the module's first week only.
- Typing a Supabase client param as a narrow structural type (e.g. `SupabaseLike`) against the real client causes "type instantiation is excessively deep." Type the param `any` with a comment instead. Same error occurs with ternaries inline in a query-builder chain — use `let q = ...; if (cond) q = q.foo()` instead.
- **Per-student analytics live under `/teacher/students/*`, not `/admin/students/*`** (moved 2026-07-07 so teachers can reach them too — `/admin/*` is admin/principal-only via `AuthGuard`, `/teacher/*` allows all three roles). Only `/admin/students/import` (roster upload) stays under admin. There are TWO independent scoring sources, both feeding the same `TagScore`/`StudentAbility`/`TagWeakness` shapes in `src/lib/analytics.ts` so they drop into the same UI: (1) daily exit-ticket tags (`buildStudentTagScores`/`groupStudentsByAbility`/`weakStudentsByTag`, keyed by `module.academic_tags`) and (2) real exam per-item results (`buildStudentIndicatorScores`/`groupStudentsByIndicatorAbility`/`weakStudentsByIndicator`, keyed by `test_items.indicator_code` + `test_item_responses`, same tap-equals-wrong convention as the tests/NT-cycle pages). Don't blend them into one number — show both, the user explicitly wants them distinguishable. When `role === 'teacher'`, the students list/groups pages auto-scope to the teacher's own `teacher_classrooms` rooms and `teachers.subjects` (admin/principal see everything, unscoped).
- **`module.academic_tags` values ARE real curriculum indicator short-codes** (e.g. "ป.3/1"), verified live against the `indicators` table — NOT free-text skill labels, and NOT classroom names, even though Thai classroom names use the exact same "ชั้น/เลข" format ("ป.3/1" = ห้อง 1) and are trivially confused with them (this confusion is what the user flagged 2026-07-08). Any UI displaying a bare tag/code MUST resolve + show the description from `indicators` (keyed `${subject}::${code}`, since the same short code is reused across different subjects with different meanings — e.g. Math "ป.3/1" ≠ Thai "ป.3/1"). All `TagScore`/`TagWeakness`-producing functions in `analytics.ts` (`buildStudentTagScores`, `buildModuleTagSummary`, `weakStudentsByTag`, `buildStudentIndicatorScores`, `weakStudentsByIndicator`) bucket by `(subject, tag)` internally for this reason — never re-introduce a bare-tag-only Map key, it silently blends two unrelated indicators' scores together. `buildModuleTagSummary()` is the shared helper for the class/school-wide tag radar (teacher/overview, admin/dashboard) — don't re-duplicate that tagMap-building loop inline again.
- **Postgres CHECK constraints silently reject writes the app-level TypeScript union allows.** `tests_type_check` never included `'formative'` even though `TestType` and the create/edit UI both did — both creating AND editing a test as `formative` failed with a raw constraint-violation alert (`supabase-system15-fix-tests-type.sql`, includes all 5 `TestType` values). Lesson: when the app has a typed enum backed by a DB CHECK constraint, they can drift independently — verify the constraint's allowed values match the TS union whenever the union changes. Also fixed in the same pass: `createTest()` in `src/app/teacher/tests/page.tsx` was silently swallowing insert errors (never checked `error`) — a failed insert looked like a no-op. Every insert/update call must check `error` and `alert()`, no silent failures.
- **Math/LaTeX-in-plaintext**: AI web tools (ChatGPT/Gemini/Claude) render LaTeX prettily in their own UI, but the copied plain text keeps raw LaTeX source (`\frac{3}{4}`, `$...$`, `\(...\)`) which is unreadable once pasted into this system's plain-text rendering. Fixed with defense in depth: `MATH_PLAIN_TEXT_RULE` (a prompt-instruction constant in `src/lib/exam-import.ts`) must be included in the "ข้อกำหนดสำคัญ" block of every prompt builder that can produce math (exam, worksheet, lesson-plan prompts all include it), plus `latexToPlainText()` sanitizer run on every paste-back parse path (`parseExamText`, lesson-plan `parseAIOutput`). Any NEW prompt-builder/paste-back feature that could touch math content must wire in both halves, not just one.
- **Teacher tools now live under `/teacher/tools/*`** (hub page + random-picker, random-group-maker, activity-timer), NOT `/admin/manage`. The random-student-picker used to live at `/tools/random` with an **unguarded layout** (`src/app/tools/layout.tsx` had no `AuthGuard` at all — a real access-control gap) — that whole `/tools` directory was deleted when the picker moved. Any new "helper tool for teachers" (the user explicitly wants more of these) belongs under `/teacher/tools/`, listed in that hub's `TOOLS` array, never a bare unguarded top-level route.
- **Dashboard filter architecture**: `admin/dashboard/page.tsx` was restructured from "compute once inside the fetch `useEffect`" to **raw fetched state + `useMemo` derivation** (`rawModules`/`rawStudents`/`rawAssessments`/`pacingMap`/`courses` are raw; `visibleModules`/`visibleStudents`/`summaries`/`tagData`/`warnings`/`barData` etc. are all `useMemo`) specifically so the new subject/room filter chips recompute instantly client-side with no extra network round-trip. Reuse this pattern for any future filterable dashboard-style page instead of refetching on filter change.
- **Teacher-overview per-lesson-plan + per-test-indicator summary** (`src/app/teacher/overview/page.tsx`): under the module bar chart there's now a per-lesson-plan list (pass/fail color by dedup'd `student_assessments` average) and a "สรุปแบบทดสอบรายตัวชี้วัด" card listing every test scoped to the teacher's subjects with overall avg% and a weakest-first per-indicator breakdown (`buildTestIndicatorStats()` in `analytics.ts`). This is where teachers see "have I assessed this lesson yet" and "which indicators are weak on this test" without leaving the overview page.
- **Assessment plans (System 16, 2026-07-13)**: teachers can now define a persistent, structured "แผนการประเมิน" per lesson plan — not just the pre-existing free-text `lesson_plans.assessment` narrative field (kept as-is, still required for the printed document). New table `assessment_plans` (one row per `lesson_plan_id`, upsert-on-conflict like `pacing_logs`) stores chosen `methods` (`exit_ticket`/`quiz`/`observation`/`homework`), `target_indicators`, and `criteria`; UI lives in `AssessmentPlanCard` (`src/components/lesson-plans/assessment-plan-card.tsx`), rendered on `/teacher/lesson-plans/[id]`, with an AI Prompt Kit to help draft criteria (`buildAssessmentPlanPrompt()` in `src/lib/assessment-plan.ts`). **Tracking is automatic, not self-reported**: the card cross-checks whether real data actually exists against the SAME `lesson_plan_id` — `student_assessments` count for exit_ticket/observation, `homework_tasks` count for homework, and `tests`/`test_scores` (via the NEW `tests.lesson_plan_id` column) for quiz — so a teacher can see "planned vs. actually happened" per lesson, not just what they said they'd do. `tests.lesson_plan_id` is optional/nullable and set at test-creation time via a subject-filtered dropdown on `/teacher/tests` — exams not tied to a specific lesson still work exactly as before.
- **Module-level pacing status must roll up from lesson-plan status.** A teacher marks "สอนจบ" per lesson-plan hour (`pacing_logs` row scoped by `lesson_plan_id`) via a DIFFERENT button than the module-wide status buttons at the top of the weekly card (module-level row, `lesson_plan_id IS NULL`) — these are two unrelated rows, so a module where every hour is individually done still read as "สอนช้ากว่าแผน" until this was fixed 2026-07-13. Both `teacher/pacing/page.tsx` (the `completedIds` Set feeding `computeSubjectPacing`) and `admin/pacing/page.tsx` (indicator coverage + `buildCrossTracking`, via a synthetic injected `PacingLog`) now also count a module complete when 100% of its lesson plans are individually marked Completed. Any NEW pacing-status aggregation must apply the same rollup — don't reintroduce a module-level-only check.
- **Daily checklist ("วันนี้ยังไม่ได้ทำอะไรบ้าง", `TodayChecklist` on `/teacher/pacing`, the teacher home page after login)** must be checked per **(lesson-plan hour, classroom)** pair, never per subject/module alone — a teacher can teach the same subject to multiple rooms, or teach several hours of the same module, and a module/subject-level check silently hides a genuinely-forgotten room or hour once ANY one is done. `assessedTodayKeys` in `teacher/pacing/page.tsx` is keyed `${lessonPlanId ?? moduleId}::${className}`. Attendance is checked separately (touch-only-exceptions model means "no row today" can mean "checked, all present" — worded softly, not as a hard warning) and rendered as its own subsection, never merged into the same list as exit-ticket/behavior recording — the user explicitly wants these treated as unrelated tasks.
- **`/teacher/assessment` must not score absent/sick/leave students.** `NOT_PRESENT_STATUSES = ['absent','sick','leave']` (not `'late'` — a late student still attended). The class-default bulk-fill and "save all" actions both exclude these students via `isNotPresentToday()`; their card shows an attendance notice instead of scoring inputs, with a manual `forceEval` override link for the rare case a teacher wants to record something anyway (e.g. work submitted from home). Existing bad rows from before this fix aren't auto-cleaned — see `supabase-cleanup-absent-scores.sql` (review the SELECT before running the DELETE).
- **Indicator short-codes collide across `standard` too, not just across subject.** Confirmed live: bare code "ม.1/1" alone maps to 8 distinct indicators under 8 different มาตรฐาน (ต 1.1, 1.2, 1.3, 2.1, 2.2, 3.1, 4.1, 4.2) for ENG_M1, each with a completely different description. This is a SEPARATE collision axis from the subject-level one already documented above — matching/grouping/display of indicators must key on `(subject, code, standard)`, not `(subject, code)`. Fixed across the whole exam pipeline (System 18, 2026-07-14): `buildExamPrompt()` tags each indicator with its มาตรฐาน + interim/final type and warns the AI explicitly when a set has colliding codes; the AI is asked to write `ตัวชี้วัด: code (มาตรฐาน X)` per item; `parseExamText()` splits that into `indicator_code` (normalized to the bare code via `normalizeIndicatorCode()`, since downstream code still compares this exact-string against `indicators.code`) and a new `standard` field (`extractStandard()`); `test_items.standard` (new column) stores it per row; `test_indicators` sync and the printed exam's "แยกตามตัวชี้วัด" answer-key summary both group by `(code, standard)` and look up the real description — NOT by bare code, which silently merges unrelated indicators. Any NEW code that groups/displays indicators by code alone should be treated as a bug.
- **Prompt Kit distribution control** (`src/components/tests/prompt-kit.tsx`): teachers can choose how the requested item count splits across ticked indicators — equal / favor ปลายทาง / favor ระหว่างทาง / custom per-indicator weight — computed via `distributeCounts()` (largest-remainder rounding, duplicated in both `prompt-kit.tsx` for the live preview and `src/lib/exam-import.ts` for the actual prompt text; keep them in sync if the algorithm changes). This is an INSTRUCTION to the AI, not an enforced constraint — the AI can still ignore it, so `ImportItems` shows a tally of actual `code × count` per indicator on paste-back so the teacher can catch a mismatch before importing, not just after printing.
